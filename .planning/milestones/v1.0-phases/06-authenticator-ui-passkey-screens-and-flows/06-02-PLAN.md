**1. form.dart — Add ContinueSignInWithFirstFactorSelectionForm**

This form is different from MFA selection (which uses radio buttons + Continue). Per user decision, it follows React's SignInSelectAuthFactor layout:

The form does NOT use the standard `fields + actions` pattern because the layout is unique (inline password + divider + passwordless buttons). Instead, override `createState()` to return a custom state that builds the layout directly.

Create a stateful form widget `ContinueSignInWithFirstFactorSelectionForm` that extends `AuthenticatorForm` (use `super._()` with empty fields and empty actions, then override the build method in the state).

The custom state should:
a. In `initState`, check `isPasskeySupported()` via `Amplify.Auth.isPasskeySupported()` and cache the result
b. Get the `ContinueSignInWithFirstFactorSelection` state from the bloc to access `availableFactors`
c. Filter factors: remove `AuthFactorType.webAuthn` if `isPasskeySupported()` returned false. Also apply `hiddenAuthMethods` from PasswordlessSettings if available (access via InheritedConfig or a new InheritedPasswordlessSettings — Claude's discretion on approach)
d. Build the layout:
   - Username display (read-only) at top — get from AuthenticatorState.username if available
   - If PASSWORD is in available factors: password TextFormField + "Sign in with password" ElevatedButton
   - If both password and passwordless methods exist: Divider with "or" text
   - For each passwordless method (EMAIL_OTP, SMS_OTP, WEB_AUTHN): full-width OutlinedButton with label from ButtonResolver (signInWithPasskey, signInWithEmail, signInWithSms). Each button immediately calls `confirmSignIn(challengeResponse: factorName)` via the AuthenticatorState/bloc
   - BackToSignInButton at bottom (existing widget)

e. Password submission: calls `confirmSignIn(confirmationValue: password)` through existing AuthenticatorState pattern
f. Passwordless button tap: calls `confirmSignIn(confirmationValue: factorName)` where factorName is the string value Cognito expects ('WEB_AUTHN', 'EMAIL_OTP', 'SMS_OTP')

Use the localized button strings from ButtonResolver (signInWithPasskey, signInWithPassword, signInWithEmail, signInWithSms) for all button labels.

**3. Loading state during passkey ceremony:**

When the user selects passkey on the factor selection screen, the form calls `confirmSignIn(challengeResponse: 'WEB_AUTHN')` via the AuthenticatorState. The existing `_confirmSignIn` method already handles the async flow — the form widget should show a loading indicator while waiting. This is handled by the existing `isPending` pattern in the form's state (no bloc changes needed for loading — the form manages its own button disabled state).

The passkey ceremony happens transparently within the state machine (Phase 2 already wired `WEB_AUTHN` challenge handling). When `confirmSignIn` is called with 'WEB_AUTHN', the state machine:
1. Sends `ANSWER: WEB_AUTHN` to Cognito
2. Receives `WEB_AUTHN` challenge with `CREDENTIAL_REQUEST_OPTIONS`
3. Platform bridge performs assertion ceremony
4. Responds to Cognito with credential
5. Returns `done` step

If the ceremony fails, the exception flows through `_exceptionController` and shows as a banner on the current screen (the factor selection screen, per UI-04 requirements). The user stays on the factor selection screen and can retry or pick another method.
